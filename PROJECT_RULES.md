# Modulr — свод правил проекта (нормативный)

Документ обязателен для агентов и людей. Нарушение правила = блокер на merge. Пересмотр правил — только через PR с записью в [CHANGELOG.md](./CHANGELOG.md).

---

## 1. Идеология (LEGO)

| ID | Правило |
|----|---------|
| I1 | **Scope** отвечает на «где/для кого»; **модуль** — на «что делаем». Модуль не содержит бизнес-ветвлений вида `if scope == "pets"`; различие — в метаданных (`scope`, `tags`, политики ядра). |
| I2 | Связь доменов **только через события** `v1.{module}.{action}` и общие контракты. Прямой импорт пакета `finance` из `tracker` (и аналоги) запрещён. |
| I3 | ИИ и оркестратор **маршрутизируют**; исполнители **подписываются** и **публикуют** исход. |
| I4 | Изоляция данных между scope по умолчанию; пересечение — явные теги, роли, подтверждение. |

Канонические scope: см. `modulr/events/segment.go` (`AllSegments`).

---

## 2. Архитектура

| ID | Правило |
|----|---------|
| A1 | Доменные модули на корневом `go.mod` (`module modulr`) используют `modulr/events` (шина, `Storage`, scope). |
| A2 | Ядро оркестрации: `core/orchestrator`, `core/aiengine`, `core/events` — отдельный контур шины ядра; интеграция с `modulr/events` через адаптер (план). |
| A3 | Реестр эндпоинтов в оркестраторе обязан знать все пары `(module, action)`, которые может выдать ИИ. |
| A4 | Пакеты `organizer/`, `telegram/`, `databases/` — отдельные `go.mod`. **Техдолг:** во всех трёх смешаны `package main` и библиотечные пакеты / есть циклы → CI опирается на **`modulr` root** как эталон; подмодули — минимум **`gofmt`**. Исправление: вынести `cmd/*`, разорвать циклы (интерфейсы, `internal/` для типов шины). |

## 3. Инженерные требования

| ID | Правило |
|----|---------|
| E1 | Публичные функции, обращающиеся к I/O, БД, HTTP, шине — принимают **`context.Context`** первым аргументом; уважают отмену. |
| E2 | Конкурентный доступ к памяти — **`sync.Mutex` / `sync.RWMutex`**; карты без гонок. |
| E3 | Ошибки не глотаются без лога/возврата; `panic` вне тестов и recovery-границ шины — недопустим. |
| E4 | **TODO** допустимы только в формате: `// TODO(modulr-<область>): <кратко>. Action: <конкретный следующий шаг>.` |
| E5 | Внешние SDK (Telegram, pgx, …) — за интерфейсами на границе модуля, где это разумно; новые зависимости — с обоснованием в CHANGELOG. |

## 4. Кодстайл (Go)

| ID | Правило |
|----|---------|
| C1 | **`go vet` + `go build ./...`** — только для корневого модуля **`modulr`**. Подмодули `organizer/`, `telegram/`, `databases/` (смешение `main` и библиотеки в одном дереве) проходят **`gofmt`** через `scripts/modulr-check.sh`; полный `go build ./...` внутри них — вне обязательств до рефакторинга layout (§2 A4). |
| C2 | Имена событий — строки `v1.module.action`; константы — в `events`/`core/events`. |
| C3 | Комментарии к экспортируемым символам — на русском или английском единообразно в пакете (в проекте преобладает русский). |

---

## 5. Взаимодействия и контракты

| ID | Правило |
|----|---------|
| V1 | Публичные интерфейсы (`*API`) не ломать без мажорной версии / записи в CHANGELOG и периода совместимости. |
| V2 | Сущности с жизненным контекстом несут **`Context`/`Scope` (segment)** и **`Tags`** согласно доменной модели. |
| V3 | Обратная связь ИИ: события `*.completed` / `v1.orchestrator.decision.outcome` — контракт для фидбека. |

---

## 6. Ограничения и безопасность

| ID | Правило |
|----|---------|
| S1 | Секреты и `.env` не коммитить (см. `.gitignore`). |
| S2 | В `aiengine` не хранить сырой PII долговременно — только метаданные решений и фидбека. |
| S3 | Логи — без токенов и паролей. |

---

## 7. Пожизненное соблюдение

1. Перед коммитом: **`./scripts/modulr-check.sh`** (или эквивалентные команды вручную).  
2. Агент Cursor: правило **`.cursor/rules/modulr.mdc`** (`alwaysApply: true`).  
3. Любое значимое изменение поведения или контракта — строка в **CHANGELOG.md**.  
4. Расхождение кода с этим документом — баг процесса; исправить код или обновить правила осознанно.
