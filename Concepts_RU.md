# Concepts.md

> **Core concepts and terminology of the AI-driven execution platform**

---

## Overview

The Go Assist platform operates on a set of well-defined concepts that form the foundation of its execution model. Understanding these concepts is essential for developing modules and integrating with the system.

---

## Action

### Definition

An **Action** represents a single, atomic operation that the system should execute. It's the fundamental unit of work generated by the AI layer and consumed by modules.

### Structure

```go
type Action struct {
    Module string                 `json:"module"`    // Target module name
    Type   string                 `json:"type"`      // Action type within module
    Params map[string]interface{} `json:"params"`    // Action parameters
    ID     string                 `json:"id"`        // Unique action identifier
    Dependencies []string         `json:"dependencies"` // Action dependencies
}
```

### Fields Explained

- **Module**: The target module that should execute this action (e.g., "finance", "calendar", "email")
- **Type**: The specific action type within the module (e.g., "create_transaction", "schedule_meeting")
- **Params**: Key-value pairs containing action-specific parameters
- **ID**: Unique identifier for tracking and dependency management
- **Dependencies**: List of action IDs that must complete before this action

### Examples

```go
// Finance transaction action
Action{
    Module: "finance",
    Type:   "create_transaction",
    Params: map[string]interface{}{
        "amount":      150.00,
        "currency":    "USD",
        "category":    "groceries",
        "description": "Weekly grocery shopping",
    },
    ID: "action_001",
}

// Calendar event action
Action{
    Module: "calendar",
    Type:   "create_event",
    Params: map[string]interface{}{
        "title":       "Team Meeting",
        "start_time":  "2024-01-15T10:00:00Z",
        "duration":    "1h",
        "attendees":   []string{"alice@company.com", "bob@company.com"},
    },
    ID: "action_002",
    Dependencies: []string{"action_001"}, // Depends on transaction
}
```

### Action Lifecycle

1. **Generation**: AI layer creates actions based on user input
2. **Validation**: Core validates actions against module registry
3. **Dispatch**: Core publishes actions to EventBus
4. **Execution**: Target module executes the action
5. **Completion**: Module returns result and marks action complete

## Result

### Definition

A **Result** represents the outcome of executing an Action. It provides feedback on success, data, and any errors that occurred during execution.

### Structure

```go
type Result struct {
    ActionID string                 `json:"action_id"` // Corresponding action ID
    Success  bool                   `json:"success"`    // Execution success status
    Data     interface{}            `json:"data"`       // Result data (if successful)
    Error    string                 `json:"error"`      // Error message (if failed)
    Metadata map[string]interface{} `json:"metadata"`   // Additional metadata
    Duration time.Duration         `json:"duration"`   // Execution time
}
```

### Fields Explained

- **ActionID**: Links this result to its corresponding action
- **Success**: Boolean indicating if execution succeeded
- **Data**: Result payload (varies by action type)
- **Error**: Error description if execution failed
- **Metadata**: Additional information (timestamps, IDs, etc.)
- **Duration**: Time taken to execute the action

### Examples

```go
// Successful transaction result
Result{
    ActionID: "action_001",
    Success:  true,
    Data: map[string]interface{}{
        "transaction_id": "txn_123456",
        "balance_after": 2850.00,
        "status":        "completed",
    },
    Metadata: map[string]interface{}{
        "created_at": "2024-01-15T09:30:00Z",
        "account_id":  "acc_789",
    },
    Duration: 150 * time.Millisecond,
}

// Failed calendar event result
Result{
    ActionID: "action_002",
    Success:  false,
    Error:    "Attendee bob@company.com is unavailable at requested time",
    Data:     nil,
    Metadata: map[string]interface{}{
        "attempted_time": "2024-01-15T10:00:00Z",
        "conflict_with": "existing_meeting_456",
    },
    Duration: 75 * time.Millisecond,
}
```

### Result Processing

1. **Generation**: Module creates result after action execution
2. **Validation**: Core validates result structure
3. **Aggregation**: Core collects results from multiple actions
4. **Reflection**: AI layer uses results for response generation
5. **Feedback**: Results feed back into AI learning system

## Input

### Definition

An **Input** represents a user request or external event that triggers the execution pipeline. It's the entry point for all system interactions.

### Structure

```go
type Input struct {
    ID          string                 `json:"id"`          // Unique input identifier
    Content     string                 `json:"content"`     // User input text
    Transport   string                 `json:"transport"`   // Source transport (http, telegram, etc.)
    Context     ExecutionContext       `json:"context"`     // Execution context
    Metadata    map[string]interface{} `json:"metadata"`    // Additional metadata
    Timestamp   time.Time              `json:"timestamp"`   // Input timestamp
}
```

### Fields Explained

- **ID**: Unique identifier for tracking
- **Content**: The actual user input text or command
- **Transport**: How the input was received (HTTP API, Telegram, scheduled job)
- **Context**: Execution context including user info, session data
- **Metadata**: Transport-specific metadata
- **Timestamp**: When the input was received

### Examples

```go
// HTTP API input
Input{
    ID:        "input_http_001",
    Content:   "Schedule a team meeting for tomorrow at 2pm",
    Transport: "http",
    Context: ExecutionContext{
        UserID:   "user_123",
        Session:  "sess_456",
        Language: "en",
        Timezone: "UTC",
    },
    Metadata: map[string]interface{}{
        "ip_address": "192.168.1.100",
        "user_agent": "Mozilla/5.0...",
    },
    Timestamp: time.Now(),
}

// Telegram bot input
Input{
    ID:        "input_tg_001",
    Content:   "Send weekly report to finance team",
    Transport: "telegram",
    Context: ExecutionContext{
        UserID:   "tg_user_789",
        ChatID:   "tg_chat_012",
        Language: "en",
        Timezone: "Europe/London",
    },
    Metadata: map[string]interface{}{
        "message_id": 12345,
        "chat_type":  "private",
    },
    Timestamp: time.Now(),
}
```

### Input Processing

1. **Reception**: Transport layer receives raw input
2. **Normalization**: Input is standardized into Input struct
3. **Validation**: Basic validation and sanitization
4. **Enrichment**: Context and metadata added
5. **Routing**: Input routed to Core orchestrator

## Context

### Definition

**Context** provides the execution environment and state information that flows through the entire pipeline. It maintains user session, system state, and execution metadata.

### Structure

```go
type ExecutionContext struct {
    UserID      string                 `json:"user_id"`      // User identifier
    Session     string                 `json:"session"`      // Session identifier
    Language    string                 `json:"language"`     // User language preference
    Timezone    string                 `json:"timezone"`     // User timezone
    Permissions []string               `json:"permissions"`  // User permissions
    Preferences map[string]interface{} `json:"preferences"`  // User preferences
    History     []ExecutionResult      `json:"history"`      // Recent execution history
    Variables   map[string]interface{} `json:"variables"`    // Context variables
}
```

### Fields Explained

- **UserID**: Unique user identifier
- **Session**: Current session identifier
- **Language**: User's preferred language
- **Timezone**: User's timezone for time-sensitive operations
- **Permissions**: List of user permissions and capabilities
- **Preferences**: User-specific preferences and settings
- **History**: Recent execution results for context
- **Variables**: Runtime variables that can be referenced in actions

### Context Propagation

Context flows through all layers of the system:

```
Input (with Context)
       |
Core Orchestrator (carries Context)
       |
AI Layer (uses Context for planning)
       |
Module Execution (receives Context)
       |
Result Generation (includes Context)
```

### Context Usage Examples

```go
// Context-aware action generation
func (ai *AIEngine) Plan(ctx context.Context, input Input) ([]Action, error) {
    // Use user timezone for time-sensitive actions
    if input.Context.Timezone != "" {
        // Adjust times based on user timezone
    }
    
    // Check permissions before suggesting actions
    if !hasPermission(input.Context.Permissions, "finance:write") {
        // Don't suggest financial actions
    }
    
    // Use history for context-aware suggestions
    if len(input.Context.History) > 0 {
        // Reference recent activities
    }
}
```

## Execution Loop

### Definition

The **Execution Loop** is the core processing cycle that transforms user input into automated actions and responses. It represents the complete lifecycle of a request through the system.

### Loop Phases

```
Input Processing
       |
AI Planning
       |
Action Validation
       |
Module Execution
       |
Result Collection
       |
AI Reflection
       |
Response Delivery
```

### Detailed Flow

#### Phase 1: Input Processing
```go
func (o *Orchestrator) ProcessInput(ctx context.Context, rawInput RawInput) (Input, error) {
    // 1. Validate and sanitize input
    if err := validateInput(rawInput); err != nil {
        return Input{}, err
    }
    
    // 2. Normalize to standard format
    input := normalizeInput(rawInput)
    
    // 3. Enrich with context
    input.Context = o.buildContext(ctx, rawInput)
    
    // 4. Log input for tracking
    o.logInput(input)
    
    return input, nil
}
```

#### Phase 2: AI Planning
```go
func (o *Orchestrator) planActions(ctx context.Context, input Input) ([]Action, error) {
    // 1. Send input to AI for planning
    planReq := PlanRequest{
        Input:   input.Content,
        Context: input.Context,
        History: input.Context.History,
    }
    
    // 2. Get AI response
    planResp, err := o.ai.Plan(ctx, planReq)
    if err != nil {
        return nil, err
    }
    
    // 3. Validate actions against registry
    for _, action := range planResp.Actions {
        if !o.registry.IsValidAction(action.Module, action.Type) {
            return nil, fmt.Errorf("invalid action: %s.%s", action.Module, action.Type)
        }
    }
    
    return planResp.Actions, nil
}
```

#### Phase 3: Action Validation
```go
func (o *Orchestrator) validateActions(ctx context.Context, actions []Action) error {
    // 1. Check action dependencies
    if err := o.validateDependencies(actions); err != nil {
        return err
    }
    
    // 2. Check user permissions
    for _, action := range actions {
        if !o.hasPermission(ctx, action) {
            return fmt.Errorf("permission denied for action: %s.%s", action.Module, action.Type)
        }
    }
    
    // 3. Validate action parameters
    for _, action := range actions {
        if err := o.validateActionParams(action); err != nil {
            return err
        }
    }
    
    return nil
}
```

#### Phase 4: Module Execution
```go
func (o *Orchestrator) executeActions(ctx context.Context, actions []Action) ([]Result, error) {
    var results []Result
    
    // 1. Create execution graph
    graph := o.buildExecutionGraph(actions)
    
    // 2. Execute actions in dependency order
    for _, level := range graph.TopologicalLevels() {
        levelResults := o.executeLevel(ctx, level)
        results = append(results, levelResults...)
    }
    
    return results, nil
}

func (o *Orchestrator) executeLevel(ctx context.Context, actions []Action) []Result {
    var wg sync.WaitGroup
    results := make(chan Result, len(actions))
    
    // Execute actions in parallel
    for _, action := range actions {
        wg.Add(1)
        go func(a Action) {
            defer wg.Done()
            result := o.executeSingleAction(ctx, a)
            results <- result
        }(action)
    }
    
    wg.Wait()
    close(results)
    
    // Collect results
    var levelResults []Result
    for result := range results {
        levelResults = append(levelResults, result)
    }
    
    return levelResults
}
```

#### Phase 5: Result Collection
```go
func (o *Orchestrator) collectResults(ctx context.Context, results []Result) ExecutionSummary {
    summary := ExecutionSummary{
        TotalActions: len(results),
        SuccessCount: 0,
        FailureCount: 0,
        Results:       results,
    }
    
    for _, result := range results {
        if result.Success {
            summary.SuccessCount++
        } else {
            summary.FailureCount++
        }
    }
    
    return summary
}
```

#### Phase 6: AI Reflection
```go
func (o *Orchestrator) reflectResults(ctx context.Context, input Input, results []Result) (string, error) {
    // 1. Prepare reflection request
    reflectReq := ReflectRequest{
        Input:   input.Content,
        Results: results,
        Actions: o.extractActions(results),
    }
    
    // 2. Send to AI for reflection
    reflectResp, err := o.ai.Reflect(ctx, reflectReq)
    if err != nil {
        return "", err
    }
    
    return reflectResp.Response, nil
}
```

#### Phase 7: Response Delivery
```go
func (o *Orchestrator) deliverResponse(ctx context.Context, input Input, response string) error {
    // 1. Format response for transport
    formatted := o.formatResponse(input.Transport, response)
    
    // 2. Deliver through appropriate transport
    switch input.Transport {
    case "http":
        return o.httpResponder.Respond(ctx, input.ID, formatted)
    case "telegram":
        return o.telegramResponder.Respond(ctx, input.Context.ChatID, formatted)
    default:
        return fmt.Errorf("unsupported transport: %s", input.Transport)
    }
}
```

### Loop Characteristics

#### Non-Blocking
- Each phase has configurable timeouts
- Failed actions don't block the entire pipeline
- Parallel execution where dependencies allow

#### Context-Aware
- User context flows through entire loop
- Timezone and language preferences respected
- Permission checks at appropriate points

#### Observable
- Each phase logs execution details
- Metrics collected for performance monitoring
- Errors tracked with full context

#### Resilient
- Retry logic for transient failures
- Fallback strategies for AI errors
- Graceful degradation when components fail

## Key Relationships

### Action-Result Mapping
Each Action produces exactly one Result:
```
Action{ID: "action_001"} -> Result{ActionID: "action_001"}
```

### Input-Context Association
Each Input carries its ExecutionContext:
```
Input{Context: ExecutionContext{...}} -> flows through entire pipeline
```

### Module-Action Routing
Actions are routed to modules based on the Module field:
```
Action{Module: "finance"} -> FinanceModule.Execute()
Action{Module: "calendar"} -> CalendarModule.Execute()
```

### Result-AI Feedback
Results feed back into AI for learning:
```
Results[] -> AI.Reflect() -> improved future planning
```

## Error Handling Concepts

### Error Classification
- **Input Errors**: Invalid user input or malformed requests
- **Planning Errors**: AI cannot generate valid actions
- **Execution Errors**: Module execution failures
- **System Errors**: Infrastructure or connectivity issues

### Error Propagation
```
Module Error -> Result{Success: false, Error: "..."} 
    -> Core collects error -> AI reflects with error context 
    -> User receives error message
```

### Recovery Strategies
- **Retry**: Transient failures with exponential backoff
- **Fallback**: Alternative actions when primary fails
- **Partial Success**: Continue with successful actions
- **User Notification**: Clear error messages and next steps

## Performance Concepts

### Latency Components
- **Input Processing**: ~1-5ms
- **AI Planning**: ~100-2000ms (depends on complexity)
- **Module Execution**: ~10-500ms per action
- **AI Reflection**: ~100-1000ms
- **Response Delivery**: ~1-10ms

### Throughput Optimization
- **Action Batching**: Group similar actions
- **Parallel Execution**: Independent actions run concurrently
- **Result Caching**: Cache frequent query results
- **AI Model Caching**: Cache AI responses for similar inputs

### Resource Management
- **Connection Pooling**: Database and external service connections
- **Memory Management**: Limit concurrent executions
- **Timeout Handling**: Prevent resource leaks
- **Circuit Breaking**: Fail fast for unhealthy components
